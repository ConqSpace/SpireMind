# 벤치마크 설계

이 문서는 SpireMind의 벤치마크를 단기 테스트와 장기 테스트로 나누어 정의한다.

현재 목표는 “LLM이 게임을 얼마나 잘하는가”를 바로 결론 내리는 것이 아니다. 먼저 같은 조건에서 같은 절차를 반복할 수 있어야 한다. 그래야 사망 원인이 LLM 판단 문제인지, 어댑터 실행 문제인지, 게임 운인지 구분할 수 있다.

## 1. 공통 원칙

벤치마크는 다음 두 질문을 분리해서 본다.

```text
어댑터 안정성:
LLM이 고른 합법 행동이 실제 게임 상태 변화로 안정적으로 이어지는가?

LLM 판단 품질:
같은 상태와 같은 합법 행동 목록을 받은 LLM이 생존과 진행에 유리한 선택을 하는가?
```

두 질문을 섞으면 결과 해석이 흐려진다. 예를 들어 첫 전투에서 죽었다고 해도, 원인은 카드 순서 판단일 수도 있고, 상태 전환 대기 실패일 수도 있고, 오래된 상태에 대한 행동 제출일 수도 있다.

따라서 모든 실행은 다음 값을 기록한다.

- STS2 버전
- SpireMind DLL 해시
- 브랜치와 커밋
- 캐릭터
- 승천 단계
- 시드
- 판단기 종류
- 모델 이름
- 추론 강도
- 프롬프트 버전
- 최대 판단 횟수
- 응답 제한 시간
- 결과 대기 시간
- 자동 단순 행동 사용 여부
- 사람 개입 여부

## 2. 단기 테스트

단기 테스트는 벤치마크 절차를 잠그기 위한 테스트다. 범위는 니오우 선택에서 첫 번째 전투 종료까지로 제한한다.

```text
니오우 선택
-> 지도에서 첫 전투 노드 선택
-> 첫 전투 진입
-> 첫 전투 종료
```

### 2.1 목적

단기 테스트는 아래 흐름이 안정적인지 확인한다.

- 니오우 화면을 올바른 phase로 내보낸다.
- 니오우 선택지가 `legal_actions`로 노출된다.
- LLM이 선택한 니오우 행동이 실제 런 시작으로 이어진다.
- 지도에서 첫 전투 노드 이동 행동이 노출된다.
- 첫 전투 진입 직후 불안정한 중간 상태가 LLM에 노출되지 않는다.
- 첫 전투 중 카드, 포션, 턴 종료 행동이 실제 게임 상태 변화로 검증된다.
- 첫 전투 종료 후 보상 또는 다음 화면으로 정상 전환된다.

이 테스트는 장기 덱 운영 능력을 평가하지 않는다. 첫 전투가 쉬운 경우에도 문제 없다. 목적은 좁은 구간에서 상태 추출, 행동 제출, 실행 검증, 로그 기록이 반복 가능한지 확인하는 것이다.

### 2.2 고정 조건

초기 단기 테스트는 아래 조건을 권장한다.

```text
테스트 ID: B0_NEOW_FIRST_COMBAT
시드 수: 1개
반복 횟수: 5회
캐릭터: Ironclad
승천: 0
시작 지점: 새 런 시작
종료 지점: 첫 전투 종료
최대 판단 횟수: 30
응답 제한 시간: 120초
결과 대기 시간: 60초
자동 단순 행동: 끔
종료 화면 자동 처리: 끔
사람 개입: setup_only 또는 none
```

시드를 하나만 쓰는 이유는 원인 분석을 쉽게 하기 위해서다. 첫 벤치마크에서 여러 시드를 섞으면 니오우 선택, 지도 구조, 첫 전투 몬스터 구성이 동시에 바뀐다. 그러면 어떤 변화가 결과를 만든 것인지 파악하기 어렵다.

### 2.3 종료 조건

단기 테스트는 다음 중 하나가 발생하면 실행을 종료한다.

- 첫 전투 승리
- 첫 전투 사망
- 니오우 선택 실패
- 지도 이동 실패
- 전투 상태 export 실패
- 행동 실행 실패
- 최대 판단 횟수 초과
- 게임 또는 브리지 중단
- 사람이 수동으로 중단

첫 전투 승리만 성공으로 보지 않는다. 첫 전투 사망도 기록 가능한 결과다. 다만 실행 실패와 게임 진행 실패는 별도 실패 유형으로 분리한다.

### 2.4 주요 지표

단기 테스트에서는 승패보다 절차 안정성을 먼저 본다.

```text
절차 지표:
- 니오우 선택 성공 여부
- 첫 전투 노드 진입 성공 여부
- 첫 전투 상태 export 성공 여부
- 첫 전투 종료 여부
- invalid_action 횟수
- stale_state 횟수
- executor_failed 횟수
- timeout 횟수

전투 지표:
- 첫 전투 승리 여부
- 잃은 체력
- 전투 턴 수
- 사용 행동 수
- 턴당 남긴 에너지
- 평균 판단 시간

일관성 지표:
- 5회 반복에서 니오우 선택이 같은가
- 5회 반복에서 첫 전투 결과가 같은가
- 5회 반복에서 잃은 체력 편차가 어느 정도인가
- 같은 상태에서 선택한 action_id가 얼마나 반복되는가
```

### 2.5 통과 기준

초기 통과 기준은 아래처럼 둔다.

```text
통과:
- 5회 중 5회 첫 전투 진입
- 5회 중 5회 첫 전투 종료
- invalid_action 0회
- executor_failed 0회
- 게임 또는 브리지 중단 0회

주의:
- stale_state가 반복 발생
- 첫 전투는 종료되지만 받은 피해 편차가 큼
- 같은 상태에서 니오우 선택이 크게 흔들림
- 응답 시간이 제한 시간에 자주 근접함

실패:
- 니오우 이후 진행 불가
- 지도 이동 실패
- 전투 export 불안정
- 합법 행동이 있는데 실행 실패
- 최대 판단 횟수 초과
```

단기 테스트를 통과하면 “LLM이 잘한다”가 아니라 “비교 가능한 작은 실험 단위가 준비되었다”고 판단한다.

## 3. 장기 테스트

장기 테스트는 LLM의 전략 판단을 보기 위한 테스트다. 단기 테스트가 안정화된 뒤에 실행한다.

장기 테스트는 한 번에 여러 시드로 넓히지 않는다. 같은 시드에서 범위를 단계적으로 늘린다.

```text
B1: 같은 시드로 1막 중반까지
B2: 같은 시드로 1막 보스까지
B3: 여러 시드의 1막 비교
B4: 전체 런 비교
```

### 3.1 B1: 같은 시드 1막 중반 테스트

목적은 전투 여러 개와 보상, 지도 선택이 이어질 때 판단이 무너지지 않는지 확인하는 것이다.

```text
시드 수: 1개
반복 횟수: 5회
종료 지점: 1막 중반 지정 층 또는 최대 판단 횟수
최대 판단 횟수: 80
```

주요 지표:

- 도달 층
- 전투별 잃은 체력
- 카드 보상 선택 수
- 보상 넘기기 수
- 지도 경로 선택
- 이벤트 진입 여부
- stale_state 회복률
- action applied 비율

이 단계에서는 보스 처치 여부보다 흐름 유지가 중요하다. 플레이어는 선택을 누적하면서 체력, 덱, 경로가 서로 영향을 주는 것을 목격한다. LLM은 그 누적 상태를 보고 다음 전투 위험을 줄이는 선택을 해야 한다.

### 3.2 B2: 같은 시드 1막 보스 테스트

목적은 같은 시드 안에서 1막 전체 판단을 반복 비교하는 것이다.

```text
시드 수: 1개
반복 횟수: 5회
종료 지점: 1막 보스 처치, 사망, 최대 판단 횟수
최대 판단 횟수: 160
```

주요 지표:

- 1막 보스 도달 여부
- 1막 보스 처치 여부
- 사망 층
- 엘리트 처치 수
- 총 체력 손실
- 보상 선택 후 덱 크기
- 상점 구매 또는 제거 선택
- 이벤트 체력 손실
- 사망 직전 5개 판단

이 단계부터 LLM 판단 품질을 본다. 특히 체력이 낮은 상황에서 방어, 휴식, 포션 사용, 위험 경로 회피가 실제 행동으로 나타나는지 확인한다.

### 3.3 B3: 여러 시드 1막 비교

목적은 특정 시드에만 맞는 결과가 아닌지 확인하는 것이다.

```text
시드 수: 10개
반복 횟수: 시드당 3회
종료 지점: 1막 보스 처치, 사망, 최대 판단 횟수
```

주요 지표:

- 1막 보스 도달률
- 1막 보스 처치율
- 평균 사망 층
- 평균 총 체력 손실
- 시드별 결과 편차
- 실패 유형 분포

이 단계에서는 기준선을 반드시 함께 둔다. 최소 기준선은 `random_legal_action`과 `defend_when_threatened_baseline`이다.

### 3.4 B4: 전체 런 비교

목적은 LLM이 장기 위험과 덱 방향을 계속 관리할 수 있는지 확인하는 것이다.

이 단계는 초기 벤치마크 범위가 아니다. 1막 벤치가 안정화된 뒤에만 진행한다.

주요 지표:

- 승리 여부
- 도달 막
- 사망 층
- 총 전투 수
- 엘리트 처치 수
- 보스 처치 수
- 덱 크기와 주요 카드
- 유물 수
- 상점 선택
- 이벤트 선택 손익
- 포션 사용 시점

## 4. 비교 대상

LLM 하나만 실행하면 결과 해석이 어렵다. 최소 비교 대상은 아래와 같다.

```text
random_legal_action:
합법 행동 중 하나를 무작위로 선택한다.
어댑터와 로그 구조가 최소한 동작하는지 확인하는 하한선이다.

defend_when_threatened_baseline:
들어오는 피해가 큰 턴에는 방어를 우선한다.
기초 생존 판단 기준선이다.

llm_current:
현재 프롬프트와 현재 모델 설정을 사용하는 LLM 판단기다.

llm_variant:
프롬프트, 모델, 추론 강도 중 하나만 바꾼 실험군이다.
```

초기에는 `defend_when_threatened_baseline`과 `llm_current`만 비교해도 된다. 무작위 기준선은 실행 구조 확인에는 유용하지만, 전략 비교에는 정보량이 낮다.

## 5. 실패 유형 태그

실패는 버리지 않는다. 실패도 벤치마크 결과다.

자동으로 붙일 수 있는 태그:

- `invalid_json`: 판단기 응답이 JSON이 아님
- `invalid_action`: `legal_actions`에 없는 행동 선택
- `stale_state`: 오래된 상태 기준 행동
- `executor_failed`: 모드 실행 실패
- `timeout`: 판단 또는 결과 대기 시간 초과
- `terminal_transition`: 행동 처리 중 종료 화면으로 전환
- `game_crash`: 게임 중단
- `bridge_crash`: 브리지 중단
- `manual_abort`: 사람이 수동 중단

분석 단계에서 붙이는 태그:

- `over_attack`: 방어가 필요한 턴에 공격을 과하게 선택
- `over_defend`: 처치 가능한 턴에 방어를 과하게 선택
- `bad_target`: 우선 처치 대상이 아닌 적을 공격
- `energy_waste`: 의미 없이 에너지를 남김
- `card_order_error`: 카드 순서 때문에 손해 발생
- `potion_missed`: 포션으로 위기 상황을 줄일 수 있었지만 사용하지 않음
- `reward_drift`: 덱 방향과 맞지 않는 보상 선택
- `path_risk_error`: 체력 상태와 맞지 않는 경로 선택

분석 태그는 관찰과 해석을 구분해서 기록한다. 예를 들어 “방어를 안 해서 죽었다”라고 바로 쓰지 않는다. 먼저 “해당 턴에 적 의도 피해가 18이었고, LLM은 방어 행동 대신 공격 행동을 선택했다”처럼 관찰 가능한 사실을 남긴다.

## 6. 로그 폴더 구조

벤치마크 실행 결과는 아래 구조를 권장한다.

```text
benchmarks/
  B0_NEOW_FIRST_COMBAT/
    scenario_config.json
    seeds.json
    deciders/
      llm_current.json
      defend_when_threatened_baseline.json
    runs/
      seed_0001/
        llm_current_run_001/
          agent_config.json
          agent_events.jsonl
          metrics.json
          post_run_report.json
          replay_manifest.json
```

`scenario_config.json`은 게임 조건을 담는다. `decider_config.json` 또는 `deciders/*.json`은 판단기 조건을 담는다. 둘을 분리해야 같은 시드에서 판단기만 바꿔 비교할 수 있다.

## 7. 확장 순서

벤치마크는 아래 순서로 확장한다.

```text
1. B0 단일 시드 니오우-첫전투 테스트를 문서 기준대로 5회 반복한다.
2. 로그 구조와 실패 태그가 분석에 충분한지 확인한다.
3. 같은 B0 조건에서 기준선 판단기를 추가한다.
4. B1로 확장해 같은 시드 1막 중반까지 본다.
5. B2로 확장해 같은 시드 1막 보스까지 본다.
6. B3에서 여러 시드 비교를 시작한다.
```

이 순서를 지키는 이유는 명확하다. 플레이어가 어떤 상황을 목격했고, LLM이 어떤 의도로 개입했으며, 그 결과 어떤 상태 변화가 발생했는지 먼저 안정적으로 기록해야 한다. 그 다음에야 장기 성능을 비교할 수 있다.
