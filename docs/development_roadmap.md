# 개발 로드맵

이 문서는 SpireMind의 개발 순서를 정의한다.

목표는 빠르게 자동 플레이를 붙이는 것이 아니다. 먼저 STS2 상태를 정확히 관찰하고, 그 다음 LLM 선택을 제한된 행동으로 실행하며, 마지막으로 실험 결과를 비교 가능한 지표로 전환한다.

## 전체 방향

```text
상태 추출
-> 행동 후보 생성
-> LLM 응답 검증
-> 행동 실행
-> 로그 저장
-> 실험 분석
```

처음 잠글 것은 전투 1개다. 지도, 보상, 상점, 이벤트는 전투 자동화가 안정화된 뒤 추가한다.

## 전략 단계와 구현 단계의 대응

구현 로드맵은 `strategic_roadmap.md`의 큰 단계에 맞춰 진행한다. 구현 단계가 전략 단계를 앞질러서는 안 된다.

| 전략 단계 | 의미 | 대응 구현 단계 |
|---|---|---|
| S0 관찰 가능성 확보 | STS2 상태를 정확히 읽는다 | R1, R2, R3 |
| S1 전투 지능 측정 | LLM의 전투 판단을 측정한다 | R4, R5, R6의 전투 지표 |
| S2 성장 판단 측정 | 카드 보상과 덱 방향을 측정한다 | R7 중 보상 선택 |
| S3 등반 판단 측정 | 지도와 위험 선택을 측정한다 | R7 중 지도/상점/휴식 |
| S4 전체 런 벤치마크 | 한 판 전체를 비교 가능한 실험으로 만든다 | R6 확장, R7 완료 후 |
| S5 보정된 AI 비교 | 계산기, 규칙, 기억의 효과를 비교한다 | 별도 R8 후보 |
| S6 학습형 정책 비교 | 강화학습/모방 정책을 비교한다 | 별도 R9 후보 |

### 전략 관문

- R3이 끝나기 전에는 LLM 성능을 평가하지 않는다. 상태와 행동 후보가 불안정하면 평가가 왜곡된다.
- R5가 끝나기 전에는 보상, 지도, 상점 선택을 열지 않는다. 전투 실패와 장기 선택 실패가 섞인다.
- R6에서 전투 지표가 안정되기 전에는 전체 런 승률을 핵심 지표로 삼지 않는다.
- R7은 한 번에 전부 열지 않는다. 보상 선택, 지도 선택, 상점 선택 순서로 연다.
- S5/S6은 1막 등반 실험이 재현 가능해진 뒤 별도 단계로 분리한다.

## R1: STS2 모드 골격

### 목표

STS2에서 로드되는 최소 C# 모드를 만든다.

### 작업

- STS2 모드 템플릿 기반 프로젝트 생성
- `sts2.dll` 참조 설정
- BaseLib/Harmony 사용 여부 결정
- 모드 manifest 작성
- 빌드 결과물을 STS2 `mods` 폴더에 복사하는 스크립트 작성
- 게임 안에서 모드 활성화 확인

### 산출물

- `src/SpireMindMod/`
- `SpireMind.dll`
- `SpireMind.pck`
- `SpireMind.json`
- `scripts/deploy_mod.*`

### 완료 기준

- STS2 설정 화면에서 SpireMind 모드가 보인다.
- STS2 로더 로그에서 SpireMind manifest 발견, DLL 로드, 초기화 완료가 확인된다.
- 모드를 켜고 새 전투에 진입해도 게임이 멈추지 않는다.

### 주요 위험

- 앞서 해보기 업데이트로 모드 로더 구조가 바뀔 수 있다.
- BaseLib 버전이 게임 버전과 맞지 않을 수 있다.

### 대응

- 첫 모드는 게임 상태를 바꾸지 않는 읽기 전용으로 만든다.
- 외부 라이브러리 사용은 필요한 만큼만 한다.

### 구현 시작 상태

- 모드 골격은 `src/SpireMindMod/`에 둔다.
- 로컬 STS2 설치 경로와 `mods` 폴더 경로는 `SpireMind.Local.props` 또는 스크립트 매개변수로만 받는다.
- `SpireMind.Local.props`는 사용자별 파일이므로 커밋하지 않는다. 예시는 `SpireMind.Local.props.example`만 제공한다.
- 빌드는 `scripts/build_mod.ps1`로 확인한다.
- 배포는 `scripts/deploy_mod.ps1`로 수행한다. 실제 STS2 설치 경로는 `-ModsDir` 또는 로컬 props에서 읽는다.
- 지금 단계의 `SpireMind.dll`은 빌드 가능한 읽기 전용 골격이다. 전투 상태 접근과 JSON 출력은 R2에서 별도 검증한다.
- `SpireMindEntrypoint`는 R1 placeholder이며, 현재 STS2 로더가 직접 호출한다고 검증되지 않았다.
- `SpireMind.pck`가 필요한 모드 로더라면 Godot export 산출물을 별도로 만들어 배포 스크립트의 `-PckPath`로 전달한다. 저장소에는 로컬 pck 산출물을 커밋하지 않는다.
- R1에서는 BaseLib와 Harmony를 사용하지 않는다. 런타임 패치가 필요하다고 확인되면 R2에서 전투 상태 접근 지점 조사와 함께 재검토한다.
- `SpireMind.json`은 기존 STS2 모드 manifest 형식으로 보고된 `author`, `has_pck`, `has_dll`, `dependencies`, `affects_gameplay` 필드를 기준으로 둔다. 로더 진입점 규칙을 확인하기 전까지 근거 없는 `entrypoint` 확장은 추가하지 않는다.
- STS2 런타임은 .NET 9 계열 어셈블리를 포함할 수 있다. 그러나 R1 모드 프로젝트는 현재 저장소 검증 가능성을 우선해 `net8.0`으로 빌드한다.
- STS2 core의 `ModInitializerAttribute`를 쓰는 모드 자체 `ModEntry` 로그는 R1에서 보류한다. 현재 로컬에는 .NET 9 SDK가 없고, STS2 v0.103.2의 `sts2.dll`이 `System.Runtime 9.0`을 참조해 `net8.0` 빌드와 충돌한다. STS2 core initializer는 .NET 9 SDK 도입 또는 대상 프레임워크 조정 결정을 한 뒤 별도 작업으로 처리한다.

### R1 수동 검증 기록

- 검증 게임 버전: STS2 v0.103.2
- 로그 파일: `C:\Users\jye00\AppData\Roaming\SlayTheSpire2\logs\godot.log`
- 모드 설정 화면 노출: 성공
- 모드 활성화: 성공
- 전투 종료까지 진행: 성공
- 특이사항: 없음
- 확인된 기존 로그: SpireMind manifest 발견, DLL 로드, 초기화 완료가 남았다.
- 모드 자체 `ModEntry` 로그: 보류. 현재 R1은 빌드 가능한 읽기 전용 골격을 우선한다.
- `SpireMindEntrypoint` 호출 여부: 미검증. R1에서는 STS2 로더 수준의 manifest 인식과 DLL 로드만 완료 사실로 본다.

## R2: 전투 상태 추출

### 목표

`combat_state.v1` 형식으로 전투 상태를 JSON 파일에 쓴다.

### 작업

- `CombatState` 접근 지점 찾기
- `PlayerCombatState`에서 손패, 뽑을 카드 더미, 버려진 카드 더미, 소멸된 카드 더미 추출
- 플레이어 체력, 방어도, 에너지 추출
- 플레이어 Power 목록 추출
- 적 목록, 체력, 방어도, Power 목록 추출
- 적 의도와 예상 피해 추출
- 유물 목록 추출
- 상태 변경 시 JSON 갱신

### 산출물

- `combat_state.v1` exporter
- 예시 JSON 파일
- 상태 추출 검증 로그

### 완료 기준

- 전투 시작, 카드 사용, 턴 종료, 적 사망 후 JSON이 갱신된다.
- 카드 더미 개수와 목록이 실제 게임과 일치한다.
- 적 의도 피해량이 화면 표시와 일치한다.
- 버프/디버프 수치가 실제 게임과 일치한다.

### 주요 위험

- 카드 객체가 손패와 더미에서 다른 wrapper 형태로 들어올 수 있다.
- Power가 버프인지 디버프인지 분류가 애매할 수 있다.
- 적 의도 피해량 계산이 실제 UI 표시와 다를 수 있다.

### 대응

- `card_id`와 `instance_id`를 분리한다.
- 분류가 애매한 Power는 `powers_unknown`으로 보낸다.
- 원본 의도 타입과 계산된 피해량을 함께 저장한다.

### R2 구현 상태

- `CombatStateHarmonyPatch`를 추가해 STS2 로더의 `Harmony.PatchAll` 경로에서 전투 방 타입을 찾는다.
- STS2 타입 직접 참조 없이 리플렉션으로 플레이어, 카드 더미, 적, 의도 원문, 유물 후보를 읽어 `combat_state.v1` JSON을 쓴다.
- 출력 경로는 `%APPDATA%\SlayTheSpire2\SpireMind\combat_state.json`이다.
- 반자동 런타임 검증은 `scripts/runtime_smoke_check.ps1`로 수행한다. `-LaunchGame`의 기본 실행은 Steam URL `steam://rungameid/2868840`을 사용하며, exe 직접 실행이 필요하면 `-LaunchMode Exe`를 명시한다. 사용자가 직접 전투에 들어가 카드 1장을 사용하고 턴 종료까지 진행한 뒤, 스크립트가 JSON 생성/갱신과 주요 필드를 반복 확인한다.
- `legal_actions`, LLM 호출, 자동 행동 실행은 R2 범위에서 제외했다.
- 미검증 항목: STS2 런타임에서 실제 패치 대상 메서드가 잡히는지, 내부 필드명이 추출 휴리스틱과 맞는지, 화면의 카드 더미/적 의도와 JSON 값이 일치하는지 확인해야 한다.

## R3: 행동 후보 생성

### 목표

현재 상태에서 실행 가능한 행동만 `legal_actions`로 만든다.

### 작업

- 손패 카드별 사용 가능 여부 확인
- 카드 대상 타입 확인
- 살아 있는 적 대상 목록 생성
- 대상 없는 카드 행동 생성
- 턴 종료 행동 생성
- 행동마다 `action_id` 생성
- 실행 전 재검증 함수 작성

### 산출물

- `legal_actions` 생성기
- 행동 검증 함수
- 행동 후보 예시 JSON

### 완료 기준

- 에너지가 부족한 카드는 행동 후보에 들어가지 않는다.
- 죽은 적은 대상 후보에 들어가지 않는다.
- 대상 없는 카드는 `target_id: null`로 생성된다.
- 모든 `action_id`가 현재 `state_id` 안에서 유일하다.

### 주요 위험

- 카드 특수 조건을 놓칠 수 있다.
- 게임 내부의 실제 사용 가능 판정과 모드 판정이 다를 수 있다.

### 대응

- 1차는 기본 카드와 일반 전투 중심으로 검증한다.
- 가능하면 게임 내부의 카드 사용 가능 판정을 호출한다.
- 판정이 불확실한 카드는 후보에서 제외하고 로그에 남긴다.

## R4: LLM 브리지

### 목표

상태 JSON과 행동 목록을 외부 LLM으로 보내고, 응답을 검증한다.

### 작업

- 로컬 브리지 프로세스 작성
- API 키 설정 파일 분리
- 요청 JSON 저장
- 응답 원문 저장
- 응답 JSON 파싱
- `state_id`와 `selected_action_id` 검증
- 시간 초과 처리

### 산출물

- LLM 브리지
- 프롬프트 템플릿
- 요청/응답 로그
- 실패 응답 샘플

### 완료 기준

- LLM 응답 JSON 파싱 성공률이 95% 이상이다.
- `selected_action_id`가 합법 행동에 포함되는 비율이 95% 이상이다.
- API 키가 로그에 남지 않는다.

### 주요 위험

- 응답 지연으로 게임 흐름이 끊길 수 있다.
- LLM이 설명은 그럴듯하지만 없는 행동을 선택할 수 있다.
- 토큰 비용이 커질 수 있다.

### 대응

- 응답 제한 시간을 둔다.
- 재질문은 1회로 제한한다.
- 처음은 완전 정보 모드로 정확도를 보고, 이후 압축 모드를 비교한다.

## R5: 전투 자동 실행

### 목표

검증된 `action_id`를 실제 게임 행동으로 실행한다.

### 작업

- 카드 사용 실행 경로 조사
- 대상 지정 실행 경로 조사
- 턴 종료 실행 경로 조사
- 실행 후 상태 갱신 대기
- 실행 실패 시 새 상태 추출
- 강제 대체 행동 적용

### 산출물

- 카드 실행기
- 턴 종료 실행기
- 실행 결과 로그

### 완료 기준

- 일반 전투 10개를 게임 충돌 없이 자동 진행한다.
- 카드 사용 후 에너지, 손패, 더미 변화가 JSON에 반영된다.
- 적 사망 후 대상 후보가 갱신된다.
- 잘못된 응답 때문에 게임이 멈추지 않는다.

### 주요 위험

- UI 입력 방식과 내부 명령 방식 중 어느 쪽이 안전한지 불확실하다.
- 카드 효과 처리 중간에 상태를 너무 빨리 읽을 수 있다.

### 대응

- 먼저 내부 명령 방식 가능성을 조사한다.
- 불안정하면 UI 입력 방식으로 우회한다.
- 실행 후 상태 안정화 대기 조건을 둔다.

## R6: 실험 러너와 분석

### 목표

여러 시드와 반복 실행 결과를 비교 가능한 지표로 저장한다.

### 작업

- 실험 설정 파일 작성
- 시드 목록 관리
- 모델 이름과 설정값 기록
- 전투별 결과 저장
- 사망 직전 5턴 자동 추출
- 요약 지표 생성

### 산출물

- `experiments/` 설정 파일
- `runs/` 로그 파일
- 실험 요약 리포트

### 완료 기준

- 고정 시드 20개를 같은 조건으로 반복 실행할 수 있다.
- 사망 층, 받은 피해, 응답 실패율, 평균 응답 시간이 집계된다.
- 기준선 AI와 LLM 결과를 같은 표로 비교할 수 있다.

### 주요 위험

- 로그가 커져서 분석이 어려워질 수 있다.
- 실험 조건이 누락되면 결과를 재현할 수 없다.

### 대응

- 로그와 요약 지표를 분리한다.
- 모든 실험에 `experiment_id`와 모델 설정을 저장한다.

## R7: 전체 등반 확장

### 목표

전투 밖 선택을 추가해 1막 등반 실험을 만든다.

R7은 전략 단계 S2와 S3에 해당한다. 한 번에 모든 비전투 선택을 열지 않는다. 먼저 보상 선택으로 성장 판단을 측정하고, 그 다음 지도와 상점으로 위험 판단을 측정한다.

### 작업

- 카드 보상 선택
- 보상 건너뛰기
- 지도 경로 선택
- 상점 카드/유물/제거 선택
- 휴식/강화 선택
- 이벤트 선택

### 산출물

- `reward_state.v1`
- `map_state.v1`
- `shop_state.v1`
- `event_state.v1`
- 1막 등반 실험 리포트

### 완료 기준

- 20개 고정 시드에서 1막 보스 도달률을 계산할 수 있다.
- 전투 실패와 비전투 선택 실패를 구분할 수 있다.
- 카드 보상 이후 덱 방향 실패를 태그로 남길 수 있다.

### 주요 위험

- 장기 선택은 실패 원인이 늦게 드러난다.
- 보상/지도/상점까지 한 번에 열면 분석이 흐려진다.

### 대응

- 전투가 안정화된 뒤 하나씩 연다.
- 보상 선택부터 추가하고, 지도와 상점은 뒤로 미룬다.

## R8 후보: 보정된 AI 비교

### 목표

LLM 단독보다 어떤 보정이 성능을 올리는지 비교한다.

이 단계는 전략 단계 S5에 해당한다. R8은 R7 이후에만 시작한다. 1막 등반 실험이 재현 가능하지 않으면 보정 효과를 신뢰할 수 없다.

### 비교 후보

- LLM 단독
- LLM + 피해 계산기
- LLM + 카드 순서 규칙
- LLM + 위험 평가 규칙
- LLM + 덱 요약 메모리
- LLM + 기준선 정책 대체 행동

### 완료 기준

- 보정 방식별 같은 시드 비교가 가능하다.
- 어떤 실패 유형이 줄었는지 지표로 확인된다.
- 보정이 LLM의 전략 판단을 가리는지 확인된다.

### 주요 위험

- 보정 규칙이 강해져서 LLM 실험이 아니라 규칙 AI 실험이 될 수 있다.

### 대응

- LLM 단독 결과를 항상 함께 저장한다.
- 보정이 개입한 행동에는 별도 로그 태그를 붙인다.

## R9 후보: 학습형 정책 비교

### 목표

강화학습 또는 모방 정책이 LLM보다 나은 구간을 찾는다.

이 단계는 전략 단계 S6에 해당한다. 처음부터 목표로 삼지 않는다. 전투 로그가 충분히 쌓인 뒤 검토한다.

### 작업

- LLM 전투 로그를 학습 데이터로 정리
- 전투 행동만 대상으로 한 정책 실험
- LLM 정책과 학습형 정책의 같은 시드 비교
- 하이브리드 구조 검토

### 완료 기준

- 학습형 정책이 전투 행동에서 LLM보다 빠르게 실행된다.
- 계산 실수와 카드 순서 실수가 줄어드는지 확인된다.
- 장기 선택은 LLM이 유지하는 하이브리드 가능성을 판단한다.

### 주요 위험

- 학습 환경 구축 비용이 커질 수 있다.
- 패치로 카드/적 구조가 바뀌면 학습 결과가 빨리 낡을 수 있다.

### 대응

- 전체 게임이 아니라 전투 행동만 먼저 학습 대상으로 삼는다.
- 학습 정책은 핵심 경로가 아니라 비교 실험 대상으로 유지한다.

## 1차 마일스톤

### M1: 읽기 전용 전투 상태 exporter

플레이어는 전투에 들어간다. 모드는 매 턴 상태 JSON을 쓴다. LLM은 아직 호출하지 않는다.

완료 기준:

- `combat_state.v1` 예시가 실제 전투에서 생성된다.
- 손패와 3개 카드 더미가 분리된다.
- 플레이어/적 체력, 방어도, Power, 적 의도가 들어간다.

### M2: LLM 응답 dry-run

모드는 상태를 보내고 LLM 응답을 받지만, 게임에 실행하지 않는다.

완료 기준:

- 응답 파싱 성공률 95% 이상
- 합법 행동 선택률 95% 이상
- 요청/응답 로그 저장

### M3: 자동 전투 10회

LLM이 고른 행동을 실제 전투에 적용한다.

완료 기준:

- 일반 전투 10개를 충돌 없이 완료한다.
- 실패 응답이 발생해도 자동 플레이가 멈추지 않는다.
- 전투 결과와 받은 피해가 저장된다.

### M4: 1막 전투 실험

고정 시드 20개에서 1막 전투 성능을 측정한다.

완료 기준:

- 평균 사망 층을 계산한다.
- 1막 보스 도달률을 계산한다.
- 주요 실패 유형 상위 5개를 뽑는다.
